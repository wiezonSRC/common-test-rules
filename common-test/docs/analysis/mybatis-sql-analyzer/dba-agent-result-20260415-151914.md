## 1단계: 쿼리 및 로직 요약

### 프로젝트 개요

`mybatis-sql-analyzer-core` + `mybatis-sql-analyzer-plugin` 은 MyBatis XML 매퍼 파일에서 동적 쿼리를 추출하고, 실제 DB에 EXPLAIN을 실행한 뒤, AI(LLM)에 전달할 쿼리 튜닝 프롬프트를 자동 생성하는 개발 보조 도구입니다.

### 핵심 클래스 역할

| 클래스 | 역할 |
|---|---|
| `SqlExtractor` | MyBatis XML에서 동적 쿼리 노드를 추출하고, `#{}`/`${}`를 `?`로 치환하여 실행 가능한 fakeSql 생성 |
| `JdbcAnalyzer` | fakeSql을 이용해 EXPLAIN 실행, JSqlParser로 테이블명 추출, JDBC MetaData API로 스키마/인덱스 정보 수집 |
| `PromptGenerator` | 위 세 가지 결과(원본 XML, fakeSql, EXPLAIN, MetaData)를 조합하여 LLM용 프롬프트 문자열 생성 |
| `SqlAnalyzerPlugin` (전체 주석 처리됨) | Gradle Task로 위 core 로직을 래핑하는 플러그인 (현재 미완성) |

### 분析 대상 핵심 쿼리 (TestMapper.xml - findBadPerformancePayments)

```sql
SELECT p.*, o.user_id, m.merchant_name
FROM payments p
JOIN orders o ON p.order_id = o.order_id
JOIN merchants m ON o.mid = m.mid
WHERE p.amount > #{minAmount}
```

- `payments`, `orders`, `merchants` 3개 테이블 JOIN
- `p.amount > #{minAmount}` : 범위 조건 (부등호)
- `<if test="status != null">` : 동적 조건으로 status 필터 추가 가능
- `SELECT p.*` : 와일드카드 조회 사용 (안티패턴)

### SampleMapper.xml 추가 분析

```sql
SELECT p.*, o.user_id, m.merchant_name
FROM payments p
JOIN orders o ON p.order_id = o.order_id
JOIN merchants m ON o.mid = m.mid
LEFT JOIN refunds r ON p.payment_id = r.payment_id
WHERE p.amount > #{minAmount}
ORDER BY p.created_at DESC
```

- 4개 테이블 JOIN (refunds 추가)
- `ORDER BY p.created_at DESC` : 인덱스 없는 컬럼 정렬

### buildFakeSql 동적 태그 처리 범위

| 태그 | 처리 방식 |
|---|---|
| `<where>` | `WHERE 1=1` + AND 절 재조합 |
| `<set>` | `SET` + 후미 콤마 제거 |
| `<foreach>` | `( ? )` 고정 치환 |
| `<choose>/<when>/<otherwise>` | `isForExplain=true` 시 첫 번째 `<when>` 선택, `false` 시 전체 순회 |
| `<include>` | `sqlSnippetRegistry`에서 FQN 조회 후 재귀 처리 |
| `<trim>` | `prefixOverrides` 기반 정리 |
| `<bind>` | 무시 (아무것도 생성하지 않음) |

### 트랜잭션 범위

이 도구는 실제 서비스 트랜잭션을 관리하지 않습니다. EXPLAIN 실행은 읽기 전용 쿼리이며, 별도의 `@Transactional` 경계가 없습니다. 분析 도구 자체의 DB 연결은 테스트에서 H2 인메모리 DB를 직접 사용합니다.

## 2단계: DB 트랜잭션 경계 분析

### 트랜잭션 설계 검토

현재 `PromptGenerator.generatePrompt()` 및 `JdbcAnalyzer` 메서드들은 `@Transactional` 경계가 전혀 없으며, Connection 객체를 외부에서 주입받는 설계입니다.

**문제점:** `JdbcAnalyzer.getExplainInfo()`와 `JdbcAnalyzer.getMetaDataInfo()`가 동일한 `Connection`을 공유하지만, 두 메서드 사이에서 커넥션 상태(autoCommit, isolation level)를 관리하는 코드가 없습니다. 실제 운영 DB에 연결할 경우, 외부에서 전달된 Connection의 격리 수준이 `REPEATABLE READ`라면 `getMetaDataInfo()`의 `SHOW VARIABLES` 쿼리와 같은 DDL 성 쿼리가 트랜잭션 내에서 실행되어 의도치 않은 락이 발생할 수 있습니다.

### 격리 수준(Isolation Level) 적합성

EXPLAIN 및 MetaData 조회는 순수 읽기 작업이므로 `READ COMMITTED` 격리 수준이 적합합니다. 현재 코드는 격리 수준을 명시적으로 설정하지 않아 호출자의 Connection 상태에 완전히 의존합니다.

### 락 전략 분析

- Pessimistic Lock(`SELECT FOR UPDATE`)은 사용하지 않습니다. (분析 도구로서 적절)
- Optimistic Lock 또한 사용하지 않습니다.
- 그러나 `getExplainInfo()` 내부에서 `ResultSet rs`를 `try` 블록 바깥에 선언하고 `finally`에서 `rs.close()`를 호출하는 패턴은, Statement가 닫히면 ResultSet이 자동으로 닫히는 JDBC 표준에 의해 이중 관리가 됩니다. 더 큰 문제는 **Statement는 try-with-resources로 관리하면서 ResultSet은 그렇지 않아** 예외 발생 시 `finally`의 `rs.close()` 자체가 NPE를 유발할 수 있는 구조입니다.

### 데드락(Deadlock) 발생 가능 시나리오

현재 코드에서는 쓰기 작업이 없으므로 데드락 리스크는 낮습니다. 그러나 플러그인이 완성되어 운영 DB에 직접 연결할 경우, `getMetaDataInfo()`의 `SHOW VARIABLES` 실행이 MariaDB/MySQL의 `information_schema` 테이블에 암묵적인 메타데이터 락을 유발할 수 있어, DDL 작업과 충돌할 가능성이 있습니다.

### sqlSnippetRegistry 스태틱 필드 위험성

```java
// SqlExtractor.java
static Map<String, String> sqlSnippetRegistry;
```

이 필드는 `static`이지만 `final`이 아니고 접근 제어자가 없습니다(package-private). 테스트에서 직접 외부에서 값을 할당하는 구조(`SqlExtractor.sqlSnippetRegistry = SqlExtractor.getSqlSnippetRegistry(...)`)는 멀티스레드 환경에서 **경쟁 조건(Race Condition)**을 유발하며, 서로 다른 테스트 케이스가 동시에 실행될 경우 레지스트리 상태가 오염될 수 있습니다.

## 3단계: EXPLAIN/ANALYZE 기반 성능 튜닝 및 인덱스 리뷰

### 3-1. 핵심 쿼리 EXPLAIN 예상 결과 (TestMapper - findBadPerformancePayments)

현재 인덱스 구성:
- `payments`: PK(payment_id), `payments_amount_idx(amount)`
- `orders`: PK(order_id), `orders_user_id_idx(user_id)`
- `merchants`: PK(mid), `idx_merchants_status(status)`
- `refunds`: PK(refund_id), `idx_refunds_payment_id(payment_id)`

```sql
SELECT p.*, o.user_id, m.merchant_name
FROM payments p
JOIN orders o ON p.order_id = o.order_id
JOIN merchants m ON o.mid = m.mid
WHERE p.amount > #{minAmount}
```

예상 EXPLAIN 분析:

| id | select_type | table | type | key | rows | Extra |
|---|---|---|---|---|---|---|
| 1 | SIMPLE | p | range | payments_amount_idx | 많음 | Using index condition |
| 1 | SIMPLE | o | eq_ref | PRIMARY | 1 | |
| 1 | SIMPLE | m | eq_ref | PRIMARY | 1 | |

**payments.order_id 컬럼에 인덱스가 없습니다.** `p.order_id = o.order_id` JOIN 조건에서 `orders` 테이블은 PK로 eq_ref 조회가 가능하지만, `payments.order_id`는 인덱스가 없어 `payments` 테이블을 amount 범위 스캔한 후 order_id로 매 행마다 랜덤 액세스가 발생합니다.

**orders.mid 컬럼에 인덱스가 없습니다.** `o.mid = m.mid` JOIN에서 `merchants`는 PK(mid)로 eq_ref이지만, `orders.mid` 컬럼 자체에 인덱스가 없어 orders 행이 많아질수록 성능이 저하됩니다.

### 3-2. SELECT * 안티패턴

TestMapper.xml, TestMapper2.xml, TestMapper3.xml, SampleMapper.xml 모두에서 `SELECT p.*` 또는 `SELECT *` 가 발견됩니다.

```xml
<!-- 현재 (안티패턴) -->
<select id="findBadPerformancePayments" resultType="map">
    SELECT p.*, o.user_id, m.merchant_name
    FROM payments p
    ...
```

대용량 트래픽 환경에서 `SELECT *`은 다음 문제를 야기합니다:
1. 불필요한 컬럼 데이터 네트워크 전송 비용 증가
2. 인덱스 커버링(Covering Index) 불가
3. 컬럼 추가/삭제 시 애플리케이션 사이드 이펙트

### 3-3. 인덱스 추가 제안 DDL

```sql
-- [1순위] payments.order_id: JOIN 핵심 컬럼, 누락된 인덱스
CREATE INDEX idx_payments_order_id ON payments (order_id);

-- [2순위] orders.mid: merchants와의 JOIN 컬럼
CREATE INDEX idx_orders_mid ON orders (mid);

-- [3순위] payments 복합 인덱스: amount 범위 + status 동적 조건 동시 최적화
-- status 조건이 선택적으로 붙는 동적 SQL 특성상, (status, amount) 순서보다
-- (amount, status) 순서가 범위 스캔 후 필터에 더 유리합니다.
CREATE INDEX idx_payments_amount_status ON payments (amount, status);

-- [4순위] payments.created_at: SampleMapper의 ORDER BY p.created_at DESC 최적화
CREATE INDEX idx_payments_created_at ON payments (created_at DESC);
```

### 3-4. N+1 쿼리 패턴 진단

현재 코드에서 N+1 패턴은 `SqlExtractor.getSqlSnippetRegistry()`에서 발생합니다.

```java
// 현재 구조: 파일별로 DocumentBuilder를 반복 재사용하지만,
// 루프 내에서 매 파일마다 TransformerFactory.newInstance()를 호출합니다.
for (File xmlFile : xmlFileList) {
    ...
    TransformerFactory transformerFactory = TransformerFactory.newInstance(); // 루프 내 반복 생성 (비효율)
    Transformer transformer = transformerFactory.newTransformer();
    ...
}
```

`TransformerFactory.newInstance()`는 SPI(Service Provider Interface) 탐색을 거치는 비교적 무거운 호출입니다. 매퍼 파일이 수십~수백 개일 경우 초기화 비용이 누적됩니다. `Transformer`는 thread-safe하지 않으므로 루프 내 생성은 맞지만, `TransformerFactory`는 루프 밖에서 한 번만 생성해야 합니다.

**개선 방안:**

```java
// 개선: TransformerFactory를 루프 밖에서 1회만 생성
TransformerFactory transformerFactory = TransformerFactory.newInstance();

for (File xmlFile : xmlFileList) {
    ...
    Transformer transformer = transformerFactory.newTransformer(); // 루프 내에서는 transformer만 생성
    transformer.setOutputProperty(OutputKeys.OMIT_XML_DECLARATION, "yes");
    ...
}
```

### 3-5. getExplainInfo ResultSet 관리 버그

```java
// 현재 코드 (버그 있음)
public static String getExplainInfo(Connection connection, String fakeSql) throws SQLException {
    ResultSet rs = null;
    try (Statement stmt = connection.createStatement()) {
        String sql = MessageFormat.format("EXPLAIN {0}", fakeSql);
        rs = stmt.executeQuery(sql);
        StringBuilder result = new StringBuilder();
        for (int i = 1; rs.next(); i++) {
            result.append(rs.getString(i)); // 매 행에서 1번 컬럼만 반복 읽음 (버그)
        }
        return result.toString();
    } finally {
        if (rs != null) rs.close(); // Statement가 닫히면 이미 닫힌 rs를 다시 close 시도
    }
}
```

**문제 1:** `rs.getString(i)`에서 `i`는 행 번호인데 컬럼 번호로 사용하고 있습니다. `rs.next()` 루프에서 `i`가 1, 2, 3...으로 증가하면 `rs.getString(2)`, `rs.getString(3)` 등으로 컬럼을 벗어나 `SQLException`이 발생합니다.

**문제 2:** try-with-resources로 `stmt`가 닫히면 `rs`도 자동으로 닫힙니다. `finally` 블록의 `rs.close()`는 이미 닫힌 ResultSet을 다시 닫으려 하며, JDBC 드라이버에 따라 예외를 던질 수 있습니다.

**개선된 getExplainInfo:**

```java
public static String getExplainInfo(Connection connection, String fakeSql) throws SQLException {
    StringBuilder result = new StringBuilder();
    String sql = MessageFormat.format("EXPLAIN {0}", fakeSql);

    try (Statement stmt = connection.createStatement();
         ResultSet rs = stmt.executeQuery(sql)) {

        int columnCount = rs.getMetaData().getColumnCount();

        while (rs.next()) {
            for (int col = 1; col <= columnCount; col++) {
                if (col > 1) {
                    result.append(" | ");
                }
                result.append(rs.getString(col));
            }
            result.append("\n");
        }
    }

    return result.toString();
}
```

### 3-6. buildFakeSql의 choose/otherwise 처리 버그

```java
case "choose":
    if (isForExplain) {
        Node targetNode = null;
        NodeList chooseChildren = child.getChildNodes();
        for (int j = 0; j < chooseChildren.getLength(); j++) {
            Node c = chooseChildren.item(j);
            if (c.getNodeType() == Node.ELEMENT_NODE) {
                if ("when".equalsIgnoreCase(c.getNodeName())) {
                    if (targetNode == null) targetNode = c; // 첫 번째 when 선택
                } else if ("otherwise".equalsIgnoreCase(c.getNodeName())) {
                    targetNode = c;
                    break; // otherwise를 발견하면 덮어씀 - 의도와 반대
                }
            }
        }
```

`otherwise`를 발견하면 `targetNode`를 덮어쓰고 break합니다. 즉, `<when>` 조건이 있더라도 `<otherwise>`가 있으면 항상 `<otherwise>`가 선택됩니다. EXPLAIN용 fakeSql 생성 시 가장 대표적인 조건(첫 번째 `<when>`)을 사용하는 것이 의도일 텐데, 현재 로직은 `<otherwise>`를 우선합니다.

**개선:**

```java
case "choose":
    if (isForExplain) {
        Node firstWhenNode = null;
        Node otherwiseNode = null;
        NodeList chooseChildren = child.getChildNodes();

        for (int j = 0; j < chooseChildren.getLength(); j++) {
            Node c = chooseChildren.item(j);
            if (c.getNodeType() != Node.ELEMENT_NODE) {
                continue;
            }
            if ("when".equalsIgnoreCase(c.getNodeName()) && firstWhenNode == null) {
                firstWhenNode = c;
            } else if ("otherwise".equalsIgnoreCase(c.getNodeName())) {
                otherwiseNode = c;
            }
        }

        // when이 있으면 첫 번째 when 우선 사용, 없으면 otherwise 사용
        Node targetNode = (firstWhenNode != null) ? firstWhenNode : otherwiseNode;
        if (targetNode != null) {
            fakeSql.append(" ").append(buildFakeSql(targetNode, true, currentNamespace, sqlSnippetRegistry)).append(" ");
        }
    } else {
        fakeSql.append(" ").append(buildFakeSql(child, false, currentNamespace, sqlSnippetRegistry)).append(" ");
    }
    break;
```

### 3-7. PromptGenerator의 NPE 위험

### 3-9. Plugin 모듈 미완성 - 참조 불일치

주석 처리된 `SqlAnalyzerTask.java`는 존재하지 않는 메서드를 호출합니다:
- `SqlExtractor.findMapperFiles()` - core 모듈에 존재하지 않음

플러그인이 활성화되면 즉시 컴파일 에러가 발생하는 상태입니다.

## 4단계: 잘한 점과 개선점

### 잘한 점

- **XML DTD 검증 비활성화**: `DocumentBuilderFactory`에서 `setValidating(false)` 및 `load-external-dtd` 비활성화를 일관되게 적용하여, 외부 네트워크 접근 없이 오프라인에서 안전하게 파싱합니다.
- **FQN 기반 sql 태그 캐싱**: `<include refid>` 처리 시 `namespace + "." + id` 형태의 완전한 FQN을 키로 사용하여, 멀티 매퍼 환경에서의 refid 충돌을 방지하는 설계가 우수합니다.
- **isForExplain 파라미터 분리**: `buildFakeSql()`의 `isForExplain` 플래그를 통해 EXPLAIN용(첫 번째 분기 선택)과 전체 분기 순회용을 하나의 메서드로 관리하는 설계가 실용적입니다.
- **`@Slf4j` 및 Lombok 활용**: 로깅에 `@Slf4j`를 사용하고 `System.out.println`을 지양하는 방향은 올바릅니다.
- **try-with-resources 부분 적용**: `Statement`, `ResultSet`, `Stream<Path>` 등에 try-with-resources를 대부분 적용하여 자원 누수를 방지하고 있습니다.
- **테스트 격리**: `@BeforeEach`/`@AfterEach`에서 H2 인메모리 DB를 생성/삭제하여 테스트 격리를 구현한 점이 좋습니다.

## 5단계: DB 성능 튜닝 관련 꼬리 질문 3개와 모범 답변

**Q1. buildFakeSql에서 `<foreach>` 태그를 `( ? )` 단 하나의 플레이스홀더로 치환하고 있습니다. 실제 운영 환경에서 IN 절에 수천 개의 값이 들어갈 때 이 방식이 EXPLAIN 결과에 어떤 영향을 미치며, 어떻게 개선해야 할까요?**

> `<foreach>` 로 생성되는 `IN (?, ?, ?, ... )` 절은 파라미터 개수에 따라 실행 계획이 달라집니다. IN 절의 파라미터 수가 적을 때는 인덱스 Range Scan으로 처리되지만, 수천 개 이상이면 옵티마이저가 Full Table Scan이 더 유리하다고 판단하여 계획을 바꿀 수 있습니다.

**A1. 모범 답변:**

현재 `( ? )` 단일 치환은 EXPLAIN 실행 자체는 가능하게 하지만, 실제 운영 시 발생하는 `IN (?, ?, ..., ?)` 수천 개 상황을 대변하지 못합니다. 두 가지 관점에서 개선이 필요합니다.

첫째, EXPLAIN 정확도 관점: `IN ( ?, ?, ?, ?, ?, ?, ?, ?, ?, ? )` 처럼 대표적인 멀티 파라미터(예: 10개)로 치환하면 옵티마이저가 IN 절 처리 비용을 더 현실적으로 추정합니다. `buildFakeSql()`에서 `<foreach>`의 `item`, `collection` 속성을 파싱하여 복수 개의 `?`를 생성하는 로직을 추가할 수 있습니다.

둘째, 실제 성능 관점: PG 도메인에서 결제 ID 목록으로 IN 조회를 하는 경우, 파라미터가 1000개를 초과하면 Oracle은 IN 절 제한(1000개)에 걸리고, MySQL/MariaDB는 `eq_range_index_dive_limit` 옵션(기본 200)을 초과하면 인덱스 통계 추정 방식이 변경되어 비효율적인 실행 계획이 나올 수 있습니다. 이를 회피하기 위해 IN 절 대신 임시 테이블 또는 VALUES ROW 조인 방식으로 대량 IN 처리를 구현하는 것이 대용량 환경에서 권장됩니다.

**Q2. `JdbcAnalyzer.getMetaDataInfo()`에서 `SHOW VARIABLES` 쿼리를 실행합니다. 이 코드가 운영 MariaDB에 연결되었을 때 어떤 권한 문제와 성능 리스크가 있을까요? 실제 PG 운영 환경에서는 어떻게 접근해야 할까요?**

> 분析 도구가 운영 DB에 직접 연결하여 EXPLAIN과 메타데이터를 조회하는 것의 보안 및 성능 리스크에 대해 질문합니다.

**A2. 모범 답변:**

권한 관점에서 `SHOW VARIABLES`는 `SUPER` 또는 `SYSTEM_VARIABLES_ADMIN` 권한을 요구하는 경우가 있으며, PG 운영 DB의 애플리케이션 계정은 보안 정책상 이 권한을 보유하지 않는 것이 일반적입니다. `DatabaseMetaData.getColumns()`, `getIndexInfo()` 역시 `information_schema` 조회를 내부적으로 실행하므로, 운영 DB에서 메타데이터 테이블에 대한 암묵적 잠금 및 캐시 경합이 발생할 수 있습니다.

성능 리스크 관점에서 `getIndexInfo()` 호출은 MariaDB의 `information_schema.STATISTICS` 테이블을 풀 스캔하는 경우가 있어, 테이블과 인덱스 수가 많은 운영 DB에서는 수 초 이상 응답 지연이 발생할 수 있습니다.

실무 권장 접근법은 두 가지입니다. 첫째, 분析 전용 읽기 전용 Replica DB(Read Replica)에 연결하여 운영 DB 부하를 완전히 분리합니다. 둘째, 이 도구는 개발/QA 환경에서만 사용하도록 정책으로 제한하고, 운영 DB 연결 정보를 확장(extension) 파라미터로 받되 프로덕션 URL 패턴이 입력되면 경고 또는 실행 차단 로직을 추가합니다. Gradle Plugin 특성상 빌드 타임 도구이므로 운영 환경 자동 연결은 설계 원칙에서 배제하는 것이 안전합니다.

**Q3. `SqlExtractor.buildFakeSql()`이 `${}` 파라미터(동적 리터럴 치환)를 `#{}` 와 동일하게 `?`로 치환합니다. `${}` 파라미터가 WHERE 절이나 ORDER BY 절에 사용될 때 발생할 수 있는 SQL 인젝션 리스크와 EXPLAIN 결과 왜곡 문제를 설명하고, DBA 관점에서 `${}` 사용 패턴을 어떻게 감지하고 통제해야 할까요?**

> MyBatis의 `${}` 동적 리터럴 치환이 EXPLAIN 정확도와 보안에 미치는 영향에 대해 질문합니다.

**A3. 모범 답변:**

`#{}` 는 PreparedStatement 바인딩 파라미터로 처리되어 SQL 인젝션 방지와 실행 계획 캐싱이 가능합니다. 반면 `${}` 는 JDBC 레벨에서 문자열 리터럴로 직접 삽입되며 두 가지 심각한 문제를 야기합니다.

SQL 인젝션 리스크: `ORDER BY ${sortColumn}` 패턴에서 외부 입력이 검증 없이 삽입되면 `ORDER BY (SELECT password FROM users LIMIT 1)` 같은 공격이 가능합니다. PG 도메인에서 결제 내역 정렬 컬럼을 `${}` 로 받는 경우 카드번호 등 민감정보 노출로 이어질 수 있습니다.

EXPLAIN 결과 왜곡: `${}` 파라미터를 `?`로 치환하면 EXPLAIN은 실제 리터럴 값을 기반으로 한 실행 계획이 아닌 파라미터 치환 계획을 반환합니다. 예를 들어 `ORDER BY ?`는 문법 오류이거나 EXPLAIN에서 Filesort 여부를 정확히 판단하지 못하게 합니다. 실제 컬럼명이 삽입되어야 옵티마이저가 해당 컬럼의 인덱스 유무를 기반으로 정렬 방식을 결정합니다.

DBA 관점의 통제 방안: 이 분析 도구에 `${}` 패턴 감지 기능을 추가하는 것이 중요합니다. XML 노드의 텍스트 값에서 `\$\{[^}]+\}` 패턴을 정규식으로 탐지하고, 발견 시 (1) 경고 메시지를 프롬프트에 포함시키고 (2) `${}` 에 들어갈 수 있는 허용 값 목록(Whitelist)을 Enum으로 제한하도록 코드 리뷰 가이드를 제공해야 합니다. `${}` 는 동적 테이블명이나 컬럼명처럼 PreparedStatement로 바인딩 불가능한 경우에만 사용하고, 그 경우에도 반드시 Whitelist 검증을 거치는 것이 PG 도메인의 보안 원칙입니다.
