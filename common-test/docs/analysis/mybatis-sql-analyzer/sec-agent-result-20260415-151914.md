## step1_code_analysis

### [코드 목적 요약]

`mybatis-sql-analyzer-core`는 MyBatis XML 매퍼 파일을 파싱하여 특정 `queryId`의 동적 쿼리를 추출하고, MyBatis 태그(`<if>`, `<where>`, `<include>` 등)를 실제 실행 가능한 fake SQL로 변환한 뒤, 연결된 데이터베이스에 `EXPLAIN`을 수행하고, 테이블 스키마·인덱스 메타데이터를 수집하여 AI 쿼리 튜닝 프롬프트를 생성하는 라이브러리다. `mybatis-sql-analyzer-plugin`은 해당 라이브러리를 Gradle 플러그인 형태로 래핑하여 `generateSqlPrompt` 태스크로 개발자가 CLI에서 직접 호출할 수 있도록 설계된 플러그인이다(현재 전체 소스가 주석 처리 상태).

**기술 스택:** Java 17, JSQLParser 4.8, MyBatis 3.5.15, Spring-JDBC 6.1.2, Lombok, JAXP(javax.xml), Gradle Plugin API.

**주요 클래스/메서드:**
- `SqlExtractor.buildFakeSql()` — MyBatis 동적 태그를 재귀 처리하여 fake SQL 생성
- `SqlExtractor.getQueryIdDetail()` / `getSqlSnippetRegistry()` — XML DOM 파싱
- `JdbcAnalyzer.getExplainInfo()` — EXPLAIN 실행
- `JdbcAnalyzer.getMetaDataInfo()` — DatabaseMetaData로 스키마/인덱스 수집
- `PromptGenerator.generatePrompt()` — AI 프롬프트 빌드 및 외부 AI 전달 준비

**데이터 민감도 분류:**
- **DB 접속 자격증명 (CRITICAL):** 플러그인 Extension에 jdbcUrl/jdbcUser/jdbcPassword 평문 관리 구조
- **내부 DB 스키마 정보 (HIGH):** 테이블명, 컬럼명, 타입, 인덱스 구조가 AI 프롬프트로 외부 전송
- **SQL 구조 정보 (HIGH):** 비즈니스 쿼리 로직 원문이 외부 AI 서비스로 전달

## step2_architecture_analysis

### [보안 및 권한 제어 구조 평가]

**레이어 분리 평가:**
이 모듈은 웹 애플리케이션이 아닌 개발 도구(Developer Tooling) 성격이므로 Controller 계층은 없다. 핵심 로직은 `SqlExtractor` → `JdbcAnalyzer` → `PromptGenerator` 순의 단방향 파이프라인으로 구성되어 있으며 레이어 간 책임 분리는 비교적 명확하다. 다만 보안 경계(입력 검증, 경로 유효성 검사)를 담당하는 계층이 전무하다.

**인증/인가 구조:**
Spring Security 등 인증 프레임워크 미적용. 이 도구는 개발자의 로컬 환경에서 실행되는 것을 전제로 설계되어 있으나, Gradle 플러그인 활성화 시 `build.gradle`에 DB 자격증명이 평문 기재될 수 있어 CI/CD 파이프라인 또는 코드 저장소에 크레덴셜이 노출될 위험이 존재한다.

**의존성 주입 방식:**
모든 메서드가 `static`으로 설계되어 인스턴스 의존성 주입 자체가 없는 구조다. 이는 테스트 용이성을 저하시키고, `SqlExtractor.sqlSnippetRegistry`가 `static` 가변 필드로 존재하여 테스트 간 상태 오염 및 스레드 안전성 문제를 야기한다.

**민감 정보 관리 구조:**
`SqlAnalyzerExtension`(현재 주석 처리)에 `getJdbcPassword()` Property가 노출되어 있으며, 이를 제한하는 `@Secret` 어노테이션, 환경변수 강제 읽기, Vault 연동 등의 보호 장치가 전혀 없다. 활성화 즉시 `build.gradle`에 DB 비밀번호 평문 기재가 강제된다.

[심각도: CRITICAL] - SQL Injection (동적 EXPLAIN)
- 위치: JdbcAnalyzer.getExplainInfo():112
- 문제점:
  MessageFormat.format("EXPLAIN {0}", fakeSql)으로 문자열을 직접 조합한 후
  Statement.executeQuery()로 실행한다.
  fakeSql은 XML 파일 경로 → DOM 파싱 → 텍스트 추출 과정을 거치지만,
  경로 입력 자체가 외부에서 제어 가능하고 XML 내용에 DROP TABLE, ; DELETE FROM 등
  다중 구문이 삽입될 경우 Statement 단위 실행으로 다중 쿼리가 실행된다.
  H2 등 일부 드라이버는 세미콜론으로 구분된 복수 쿼리를 허용한다.
- 취약 코드 스니펫:
  String sql = MessageFormat.format("EXPLAIN {0}", fakeSql);
  rs = stmt.executeQuery(sql);
- 권고 조치:
  1. JSQLParser로 fakeSql이 SELECT 유형인지 타입 검증 후 비SELECT는 EXPLAIN 거부
  2. 세미콜론, DDL 키워드(DROP/CREATE/ALTER/TRUNCATE) 포함 여부 블랙리스트 검사
  3. DB 계정에 SELECT 전용 권한만 부여 (최후 방어선)
- 수정 코드 예시:
  public static String getExplainInfo(Connection connection, String fakeSql) throws SQLException {
      // SELECT 구문만 EXPLAIN 허용
      try {
          net.sf.jsqlparser.statement.Statement parsed = CCJSqlParserUtil.parse(fakeSql);
          if (!(parsed instanceof net.sf.jsqlparser.statement.select.Select)) {
              throw new IllegalArgumentException("EXPLAIN은 SELECT 구문에만 허용됩니다.");
          }
      } catch (JSQLParserException e) {
          throw new IllegalArgumentException("유효하지 않은 SQL 구문입니다.", e);
      }
      // 이후 기존 로직 유지
  }
- 관련 OWASP: A03:2021 – Injection
- SonarQube 규칙: squid:S2077 (Use of untrusted data in SQL query)
```

[심각도: CRITICAL] - XXE (XML External Entity Injection) — 불완전 방어 3개소
- 위치:

  SqlExtractor.getSqlSnippetRegistry():199
  SqlExtractor.buildFakeSql() include 처리:127
- 문제점:
  세 곳 모두 DocumentBuilderFactory에 load-external-dtd 비활성화만 적용되어 있다.

- 취약 코드 스니펫 (getSqlSnippetRegistry 예시):

  dbf.setValidating(false);
  dbf.setFeature("http://apache.org/xml/features/nonvalidating/load-external-dtd", false);
  // FEATURE_SECURE_PROCESSING, external-*-entities 설정 누락
- 권고 조치:
  공통 보안 팩토리 메서드를 추출하여 5개 속성을 일괄 적용.
  DOCTYPE을 허용해야 하는 경우 disallow-doctype-decl은 false 유지하고
  external-entities 2개를 false로 설정하는 방식으로 보완.
- 수정 코드 예시:
  private static DocumentBuilderFactory createSecureFactory() throws ParserConfigurationException {
      DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance();
      dbf.setValidating(false);
      dbf.setFeature("http://apache.org/xml/features/nonvalidating/load-external-dtd", false);
      dbf.setFeature("http://xml.org/sax/features/external-general-entities", false);
      dbf.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
      dbf.setFeature(javax.xml.XMLConstants.FEATURE_SECURE_PROCESSING, true);
      dbf.setAttribute(javax.xml.XMLConstants.ACCESS_EXTERNAL_DTD, "");
      dbf.setAttribute(javax.xml.XMLConstants.ACCESS_EXTERNAL_SCHEMA, "");
      return dbf;
  }
- 관련 OWASP: A05:2021 – Security Misconfiguration (XXE)
- SonarQube 규칙: squid:S2755 (XML parsers should not be vulnerable to XXE)

```
[심각도: MEDIUM] - EXPLAIN 결과 컬럼 인덱스 오용 버그
- 위치: JdbcAnalyzer.getExplainInfo():115
- 문제점:
  for(int i = 1; rs.next(); i++) { result.append(rs.getString(i)); }
  변수 i는 행(row) 카운터이지만 rs.getString(i) 에서 컬럼 인덱스로 오용된다.
  EXPLAIN 결과가 여러 컬럼인 경우(MySQL EXPLAIN은 12개 컬럼 반환),
  두 번째 행부터는 rs.getString(2), rs.getString(3) ... 으로 엉뚱한 컬럼이 읽힌다.
  10개 컬럼인 DB에서 i=11이 되면 SQLException이 발생한다.
  PromptGenerator를 통해 AI에 전달되는 EXPLAIN 정보가 오염되어 잘못된 튜닝 제안을 유발한다.
- 취약 코드 스니펫:
  for(int i = 1; rs.next(); i++){

## step4_pros_and_cons

### [안전성 측면의 장단점]

**잘 구현된 보안 요소 (Pros):**

1. **DTD 외부 로딩 차단 (부분 적용):** 모든 DocumentBuilderFactory 생성 지점에서 일관되게 `setFeature("http://apache.org/xml/features/nonvalidating/load-external-dtd", false)` 가 적용되어 있다. XXE 방어의 첫 번째 레이어는 갖춰진 상태이며, 외부 DTD 원격 접속을 통한 기본적인 SSRF는 차단된다.

2. **try-with-resources 일관 적용:** `JdbcAnalyzer.getMetaDataInfo()`, `SqlExtractor.getSqlSnippetRegistry()` 등에서 ResultSet, Statement, Stream 등 자원이 try-with-resources로 안전하게 닫힌다. JDBC 자원 누수로 인한 DB 연결 고갈(DoS) 위험이 낮다.

3. **MyBatis `${}` 치환 처리:** `buildFakeSql()`에서 `#{}`와 `${}` 모두를 `?`로 치환하는 로직(`replaceAll("[#$]\\{[^}]+\\}", "?")`)이 구현되어 있어, 생성된 fake SQL이 파라미터 바인딩 형태로 처리된다. MyBatis `${}` 직접 치환으로 인한 SQL Injection이 fake SQL 단계에서는 방지된다.

4. **유틸리티 클래스 인스턴스화 방지:** `JdbcAnalyzer`, `SqlExtractor`, `PromptGenerator` 모두 `private` 생성자를 선언하여 인스턴스화를 차단하고 있다. 유틸리티 클래스 패턴의 올바른 적용이다.

5. **Lombok @Slf4j 로깅 체계:** 프로덕션 코드(`SqlExtractor`, `JdbcAnalyzer`)에서 `@Slf4j`를 사용하여 표준 로깅 체계를 준수한다. `System.out.println`은 테스트 코드에만 남아 있어 프로덕션 코드 노출은 없다.

## step5_interview_qna

### [시큐어 코딩 및 스레드 안전성 관련 꼬리 질문 3개와 모범 답변]

**Q1. getExplainInfo() 메서드에서 MessageFormat.format("EXPLAIN {0}", fakeSql)으로 EXPLAIN을 실행하고 있는데, 왜 PreparedStatement를 사용하지 않았을까요? 그리고 이 구조에서 SQL Injection을 어떻게 방어할 수 있을까요?**

A1.
- **핵심 개념 설명:** `EXPLAIN`은 DDL 문의 일종이며, JDBC `PreparedStatement`는 DML 파라미터 바인딩만 지원한다. `EXPLAIN ?`처럼 PreparedStatement에 SQL 자체를 파라미터로 넘기는 방식은 지원되지 않으므로, EXPLAIN 실행 시 문자열 결합이 불가피한 구조적 제약이 있다.

- **이 코드에 적용했을 경우의 예시:**
  ```java
  // 파서 기반 검증 레이어 추가
  net.sf.jsqlparser.statement.Statement parsed = CCJSqlParserUtil.parse(fakeSql);
  if (!(parsed instanceof net.sf.jsqlparser.statement.select.Select)) {
      throw new IllegalArgumentException("EXPLAIN은 SELECT 구문에만 허용됩니다.");
  }
  // 멀티 구문 방어: 세미콜론 포함 여부 검사
  if (fakeSql.contains(";")) {
      throw new IllegalArgumentException("다중 구문이 포함된 SQL은 허용되지 않습니다.");
  }
  ```

- **관련 Java/Spring 구현 방법:**
  PreparedStatement 외에 SQL Injection 방어 수단으로는 (1) 화이트리스트 기반 파서 검증(JSQLParser `instanceof` 체크), (2) 블랙리스트 키워드 필터(DROP/CREATE/ALTER/TRUNCATE/EXEC), (3) DB 계정 최소 권한(SELECT only) 3계층 방어가 권장된다. 이 코드처럼 fakeSql이 내부에서 생성되는 경우라도 XML 파일 경로가 외부 입력이므로 Path Traversal + XXE를 통한 간접 SQL 조작 경로를 차단하는 것이 선행 조건이다.

---

**Q2. SqlExtractor.sqlSnippetRegistry가 static 가변 필드로 존재하고 테스트 코드에서 직접 외부 주입(SqlExtractor.sqlSnippetRegistry = ...)하고 있습니다. Gradle 병렬 테스트 환경에서 이것이 왜 위험하며, 어떻게 개선해야 할까요?**

A2.
- **핵심 개념 설명:** Java의 `static` 필드는 JVM 내 클래스 로더 범위에서 단일 인스턴스로 공유된다. JUnit 5의 `@TestMethodOrder` + Gradle 병렬 테스트(`maxParallelForks > 1`) 환경에서는 여러 테스트 스레드가 동시에 `SqlExtractor.sqlSnippetRegistry`를 읽고 쓸 수 있다. `HashMap`은 스레드 비안전(thread-unsafe) 자료구조이므로 `ConcurrentModificationException` 또는 무한 루프(버킷 링크드리스트 순환 참조, Java 7 이하)가 발생할 수 있다.

- **이 코드에 적용했을 경우의 예시:**
  ```java
  // 잘못된 패턴 (현재 코드)
  static Map<String, String> sqlSnippetRegistry; // static 가변 필드
  // 테스트: SqlExtractor.sqlSnippetRegistry = getSqlSnippetRegistry(...)
  
  // 올바른 패턴: 필드 제거, 파라미터로만 전달
  public static String buildFakeSql(
          Node nodeList, boolean isForExplain,
          String currentNamespace,
          Map<String, String> sqlSnippetRegistry) { // 파라미터로만 관리
      // ...
  }
  ```

- **관련 Java/Spring 구현 방법:**
  Gradle 테스트 격리를 위해 `@TestInstance(Lifecycle.PER_METHOD)`(기본값) + `@BeforeEach`에서 로컬 Map을 생성하여 메서드 파라미터로 전달하는 패턴이 스레드 안전하다. 캐시가 필요한 경우 `ConcurrentHashMap` + `volatile` 또는 `AtomicReference<Map<String, String>>`로 교체하되, 이 도구는 단일 태스크 실행이 목적이므로 static 캐시 자체가 불필요하다.

---

**Q3. PromptGenerator.generatePrompt()는 DB 스키마 메타데이터(테이블명, 컬럼명, 인덱스 구조)를 원문 그대로 외부 AI 서비스에 전송합니다. 금융/PG 도메인에서 이것이 어떤 컴플라이언스 위반을 야기하며, 기술적으로 어떻게 정보를 익명화하여 전송할 수 있을까요?**

A3.
- **핵심 개념 설명:** 금융보안원 ISMS-P 인증 기준(2.4.7 정보자산 반출입 통제) 및 PCI DSS v4.0 Requirement 3(민감 데이터 보호)에 따르면 카드 데이터 관련 DB 구조(테이블명, 컬럼명)는 내부 시스템 정보 자산으로 분류되어 사전 승인 없는 외부 전송이 제한된다. 특히 AI SaaS 서비스(ChatGPT, Gemini)의 데이터 학습 약관에 따라 전송된 정보가 모델 재학습에 사용될 수 있어 기업 정보 유출 경로가 된다.

- **이 코드에 적용했을 경우의 예시:**
  ```java
  // 메타데이터 익명화 유틸리티
  public static String anonymize(String metaDataInfo) {
      // 테이블명: PAYMENTS → T_1, ORDERS → T_2 (알파벳 순 매핑)
      // 컬럼명: PAYMENT_ID → C_1, ORDER_ID → C_2
      // 타입/크기는 유지 (튜닝에 필요)
      // DB 제품명/버전 정보는 제거
      AtomicInteger tableIdx = new AtomicInteger(1);
      AtomicInteger colIdx = new AtomicInteger(1);
      // 정규식 기반 치환 로직 구현
  }
  // PromptGenerator에서 사용
  prompt.append("[MetaData] : ").append(anonymize(metaDataInfo)).append("\n");
  ```

- **관련 Java/Spring 구현 방법:**
  (1) 익명화 매핑 테이블을 프롬프트 생성 시 별도로 출력하여 AI 응답을 역매핑 가능하게 한다. (2) `extension.isAllowSchemaTransfer()` 설정 플래그로 전송 여부를 사용자가 명시적으로 허용하도록 강제한다. (3) 장기적으로는 로컬 LLM(ollama + codellama) 사용을 권장하고, 외부 AI 전송을 선택 옵션이 아닌 비기본값으로 설정하는 것이 보안 설계 원칙(Secure by Default)에 부합한다.
